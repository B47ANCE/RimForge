# Competitive Context

RimForge exists in an ecosystem that includes RimPy, RimSort, Steam Workshop collections, manual `ModsConfig.xml` editing, and specialized diagnostic tools.

This document is intentionally high level. It guides product positioning; it is not a claim that every external capability or implementation detail is permanently current.

## User expectations established by existing tools

A credible RimWorld modpack tool is expected to provide:

- Mod discovery and profile/load-order editing
- Dependency and incompatibility warnings
- Automated or assisted sorting
- Steam Workshop integration
- Import/export and profile persistence
- Texture optimization utilities
- Clear handling of Core and official DLC
- Practical diagnostics for large mod lists

RimForge 1.0 should meet those baseline expectations wherever they align with safe, maintainable architecture.

## RimForge differentiation

RimForge is designed as a **modpack engineering platform** rather than only a sorter or list editor.

Differentiating capabilities include:

- Shared Forge Evidence consumed consistently across the application
- Explainable tri-hybrid sorting with provenance and confidence
- Dependency Map as an interactive engineering workspace
- Profile-scoped Forge DNA, issues, repair plans, and readiness review
- Atomic recovery, undo, and explicit safety boundaries
- Optional Runtime Companion diagnostics
- Unified selection, search, activity, and navigation across workflows
- Extensible curated knowledge with local-first operation

## Product rule

Competition is a parity floor, not an architecture specification. RimForge should adopt valuable user-facing outcomes without copying unsafe behavior, opaque decisions, tightly coupled designs, branding, or implementation details.

Comparisons in public documentation should remain factual, respectful, dated when necessary, and supported by reproducible evidence.
